# OpenHIS — Master Transformation Plan

This is the definitive, actionable plan synthesizing everything from this thread. It covers what to build, what to change, what to retire, and in what order — based on a full reading of every service, the docker-compose, nginx config, and all key source files.

***

## North Star Objective

Transform OpenHIS from a collection of loosely coupled Docker services into a **cohesive, profile-driven Health Information Platform** where:

- Any combination of clinical modules can be deployed with a single command
- All services are managed, monitored, and configured from one Admin plane
- Adding a new module (open-source app or custom service) follows a documented, repeatable pattern
- The architecture scales from a single-server clinic to a multi-node hospital without rewriting code

***

## The Three Governing Principles

Every decision in this plan follows three rules:

1. **Integration over reimplementation** — use mature open-source apps (OpenMRS, OpenELIS, Odoo) for clinical domain logic; build only what doesn't exist
2. **Contracts over direct calls** — no service calls another directly; all cross-service communication goes through the FHIR Bridge or Event Bus
3. **Platform-first** — the Admin plane, Compose profiles, and service registry are built before adding features to individual services

## Phase 0 — Structural Foundation

**Goal:** Zero functional change. Only reorganize. No service code is touched.

- [ ] Move `docker-compose.yml` → `compose/` directory; split into `base.yml` + 5 profile files
- [ ] Move `services/ehr`, `services/lis`, `services/pharmacy` → `services-legacy/`; create `compose/profiles/legacy.yml` that still runs them for development reference
- [ ] Create `.env` variable `OPENHIS_PROFILES` consumed by the root Compose file
- [ ] Add `redis:7-alpine` to `base.yml` (no service uses it yet, but it anchors Phase 2)
- [ ] Add `postgres` database creation for `openelis`, `odoo`, `openhis_keycloak` in `infra/postgres/init-databases.sh`
- [ ] Update nginx to use the Docker DNS resolver more robustly (`resolver 127.0.0.11 valid=10s`) — already partially there

**Outcome:** Stack still works identically. You can now run `OPENHIS_PROFILES=imaging,emr docker compose up` as a target to reach.

## Phase 1 — Identity \& Service Registry

**Goal:** Single sign-on across all services. All services know which others are active.

### 1a — Keycloak SSO

- [ ] Add Keycloak 24 container to `base.yml`
- [ ] Create `infra/keycloak/openhis-realm.json` — export with: roles (`admin`, `clinician`, `radiologist`, `lab-tech`, `pharmacist`, `patient`), OIDC clients for each service
- [ ] Replace `services/admin/security.py` session-based auth with JWT validation against Keycloak
- [ ] Configure OpenMRS OIDC login (supported natively in O3 via `oidc-authenticator` module)
- [ ] Wire Odoo OAuth2 to Keycloak
- [ ] Wire OpenELIS LDAP/OIDC to Keycloak
- [ ] All native services (RIS, HL7, AI Controller, Analytics, MPI) validate Bearer tokens against Keycloak's JWKS endpoint


### 1b — Service Registry

- [ ] Create `platform/registry.json` — source of truth for which services are active and their URLs
- [ ] OPM writes this file on every `enable`/`disable` command
- [ ] Each native service reads registry at startup; gracefully skips HTTP calls to inactive services instead of crashing
- [ ] Replace all hardcoded `*_URL` environment variables with registry lookups

**Outcome:** One login for everything. Services survive partial deployments without crashing.

## Phase 2 — Compose Profiles + OPM CLI

**Goal:** Operator can deploy any combination of modules with one command.

### 2a — Profile Engine

Each profile YAML carries a metadata block:

```yaml
# compose/profiles/laboratory.yml
x-openhis:
  profile: laboratory
  display_name: "Laboratory Suite"
  requires: [base]
  integrates_with: [emr]
  databases: [openelis]
  nginx_routes:
    - { path: /openelis, upstream: "openelis:8080" }
    - { path: /lis,      upstream: "lis-adapter:8004" }
```

- [ ] Write all 5 profile YAMLs with metadata blocks
- [ ] Write `platform/nginx_gen.py` — reads active profile metadata, renders `infra/nginx/nginx.conf.j2`, sends `nginx -s reload`


### 2b — OPM CLI (`platform/opm.py`)

- [ ] `opm init` — first-time wizard: choose profiles, set passwords, generate `.env`
- [ ] `opm enable <profile>` — validates dependencies, starts containers, updates registry, regenerates nginx
- [ ] `opm disable <profile>` — stops containers, removes routes, updates registry
- [ ] `opm status` — prints live health table of all services in active profiles
- [ ] `opm upgrade <profile>` — pulls latest images, does rolling restart

```
- [ ] `opm config set <service> <key> <value>` — writes to centralized config store in Admin DB
```

- [ ] `opm add-service <name>` — scaffolds a new native service from template (see Phase 4)

**Outcome:** `opm enable emr laboratory` is a complete, working deployment command.

## Phase 3 — FHIR Bridge Refactor + MPI as Spine

**Goal:** All cross-application data flows through defined contracts. No service calls `ehr` or `lis` directly anymore.

### 3a — FHIR Bridge → Adapter Hub

Refactor `services/fhir-bridge/` into:

```
fhir-bridge/
├── adapters/
│   ├── openmrs.py      polls /ws/fhir2/R4/ → publishes to Redis bus
│   ├── openelis.py     polls OpenELIS REST → FHIR DiagnosticReport
│   ├── odoo.py         XML-RPC → FHIR MedicationRequest / Invoice
│   └── orthanc.py      moves existing plugin.py logic here
└── events/             existing handlers → now Redis consumers
```

- [ ] Replace direct `EHR_URL` calls in FHIR Bridge with OpenMRS FHIR R4 adapter
- [ ] Replace `LIS_URL` calls with OpenELIS adapter
- [ ] All events now published to Redis Streams (`openhis:events` stream)
- [ ] Each consumer service (Analytics, HL7, MPI, AI Controller) subscribes via Redis consumer groups


### 3b — MPI as Cross-Reference Spine

- [ ] Add `sync` endpoints for OpenMRS patient IDs → `OPENMRS_ID`
- [ ] Add `sync` endpoints for OpenELIS patient IDs → `OPENELIS_ID`
- [ ] Add `sync` endpoint for Odoo partner IDs → `ODOO_PARTNER_ID`
- [ ] MPI `crossref` table already exists  — populate it from adapter events
- [ ] All services resolve patient identity through `GET /mpi/api/crossref?system=OPENMRS_ID&value=<id>`


### 3c — Analytics Collector Rewrite

- [ ] Replace all `EHR_URL/patients` calls with OpenMRS FHIR R4 queries
- [ ] Replace `LIS_URL/orders` with OpenELIS REST queries
- [ ] Replace `EHR_URL/billing` with Odoo financial API queries
- [ ] Keep the snapshot engine and export routers as-is  — only `collector.py` changes

**Outcome:** Legacy `ehr`, `lis`, `pharmacy` services are fully decoupled. Any host application can be swapped without touching other services.

## Phase 4 — Admin Plane Promotion

**Goal:** The Admin service becomes the full OPM Web UI — the single pane of glass for the entire platform.

The Admin service already has routers for auth, users, config, services, and audit . Extend it with:

### New Admin Sections

| Section | What it does | New code needed |
| :-- | :-- | :-- |
| **Platform Map** | Live topology: active profiles, service health, inter-service dependency graph | `routers/platform.py` |
| **Profile Manager** | Enable/disable profiles with pre-flight check; shows RAM estimate per profile | `routers/profiles.py` |
| **Service Config** | Centralized env var editor; replaces manual `.env` editing | extend `routers/config.py` |
| **Identity** | Keycloak-backed user CRUD + role assignment across all apps | `routers/identity.py` |
| **Event Stream** | Live Redis stream consumer shown as feed in UI | `routers/events.py` |
| **Module Catalog** | Registry of available profiles/modules with enable button | `routers/catalog.py` |

- [ ] Replace SQLite health polling (currently hardcoded `*_HEALTH` env vars ) with registry-driven polling
- [ ] Admin sessions → Keycloak OIDC (dogfoods the SSO from Phase 1)

## Phase 5 — New Service Development Contract

**Goal:** Anyone can add a new custom service in a documented, consistent way.

### The Service Contract

Every native OpenHIS service must satisfy:

```
services/<name>/
├── Dockerfile
├── requirements.txt
├── main.py              FastAPI app; reads ROOT_PATH, validates JWT from Keycloak
├── database.py          uses shared PostgreSQL (not SQLite)
├── routers/             business logic
├── static/index.html    minimal UI
├── openhis.service.json service manifest (port, profile, nginx route, bus topics)
└── tests/
```

The `openhis.service.json` manifest:

```json
{
  "name": "my-service",
  "port": 8020,
  "profile": "my-profile",
  "nginx_path": "/my-service",
  "bus": {
    "publishes": ["my-service.event-created"],
    "subscribes": ["mpi.patient-created", "fhir-bridge.order-placed"]
  },
  "depends_on": ["mpi", "fhir-bridge"]
}
```

- [ ] Write `docs/service-contract.md` — the complete guide
- [ ] Implement `opm add-service <name>` scaffold generator — creates the directory skeleton, registers in registry, adds nginx route, creates Keycloak client
- [ ] Write `docs/adapter-contract.md` — guide for wrapping a new host application (OpenELIS pattern as reference implementation)
- [ ] Write `docs/profile-contract.md` — guide for creating a new deployment profile

## What You Should Not Build

Being explicit about scope boundaries is as important as the plan itself:

- ❌ **No custom message broker** — Redis Streams is sufficient; Kafka/RabbitMQ adds ops burden with no benefit at this scale
- ❌ **No Kubernetes yet** — Docker Compose profiles map directly to Swarm stacks when you need multi-node; Helm charts come after the platform is stable
- ❌ **No rewrite of OpenMRS/OpenELIS internals** — configure and adapt, never fork
- ❌ **No custom DICOM server** — Orthanc is production-grade; your AI Controller wraps it cleanly already
- ❌ **No custom identity provider** — Keycloak is the standard; don't extend `security.py` further
